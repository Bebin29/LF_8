# Open-Source vs. Closed-Source Software: Analyse

## 1. Unterschied Open-Source und proprietäre Software

Der grundlegende Unterschied zwischen Open-Source und proprietärer Software liegt im Zugang zum Quellcode. Bei Open-Source Software ist der Quellcode öffentlich zugänglich. Jeder kann ihn einsehen, verwenden, modifizieren und — im Rahmen der jeweiligen Lizenz wie MIT, GPL oder Apache 2.0 — weiterverbreiten. Bekannte Beispiele sind Linux, Python, Firefox oder PostgreSQL. Die Entwicklung wird häufig von einer globalen Community getragen, die gemeinsam Fehler behebt, neue Funktionen entwickelt und Sicherheitslücken aufdeckt. In der Regel ist Open-Source Software kostenlos nutzbar, wobei Kosten für professionellen Support oder Hosting anfallen können.

Proprietäre Software hingegen ist Software, deren Quellcode geheim gehalten wird und im Besitz des Herstellers verbleibt. Die Nutzung wird durch kostenpflichtige Lizenzen geregelt, die entweder einmalig, als Abonnement oder nutzungsbasiert abgerechnet werden. Der Hersteller ist allein verantwortlich für Support, Updates und Bugfixes. Nutzer haben keine Möglichkeit, den Code selbst einzusehen oder anzupassen, was eine direkte Abhängigkeit vom Hersteller erzeugt. Beispiele hierfür sind Microsoft Office, Adobe Creative Suite oder SAP.

Ein weiterer wesentlicher Unterschied zeigt sich in der Sicherheitsphilosophie. Open-Source verfolgt den Ansatz "Many Eyes" — je mehr Menschen den Code prüfen, desto wahrscheinlicher werden Sicherheitslücken entdeckt. Proprietäre Software setzt dagegen auf "Security through Obscurity", also Sicherheit durch Geheimhaltung des Codes. Beide Ansätze haben ihre Berechtigung, wobei die Open-Source-Community in den letzten Jahrzehnten gezeigt hat, dass Transparenz oft zu robusterer Software führt.

## 2. Verwendete Infrastruktur und Lock-in-Effekte

Unser Projekt setzt überwiegend auf Open-Source-Technologien. Python als Programmiersprache, psutil als Systembibliothek, Git als Versionsverwaltung sowie pytest, flake8, mypy und bandit als Qualitätssicherungstools sind allesamt quelloffen und unter permissiven Lizenzen verfügbar. Diese Komponenten können jederzeit auf beliebigen Systemen betrieben werden und erzeugen keinerlei Abhängigkeit von einzelnen Herstellern. Sollte eines dieser Tools eingestellt werden, stehen zahlreiche Alternativen bereit oder die Community kann das Projekt weiterführen (Fork).

Der kritische Punkt in unserer Infrastruktur ist GitHub. Unser Repository-Hosting, die Projektverwaltung über GitHub Issues und Projects sowie die gesamte CI/CD-Pipeline über GitHub Actions sind proprietäre Dienste von Microsoft. Zwar basiert die Grundlage — Git — auf Open-Source, sodass der Code und die gesamte Versionshistorie jederzeit zu einem anderen Anbieter wie GitLab oder Bitbucket migriert werden können. Die darüber hinausgehenden Dienste sind jedoch GitHub-spezifisch. Die Workflow-Dateien unserer CI/CD-Pipeline verwenden eine proprietäre YAML-Syntax, die bei einem Wechsel zu GitLab CI, Jenkins oder CircleCI komplett neu geschrieben werden müsste. Ebenso müssten Issues, Projects und deren Verknüpfungen manuell oder per API-Export migriert werden.

Dieser Lock-in-Effekt ist ein typisches Beispiel dafür, wie auch bei überwiegender Open-Source-Nutzung eine einzelne proprietäre Komponente eine erhebliche Abhängigkeit erzeugen kann. Die Kosten eines Wechsels steigen mit der Projektgröße und der Nutzungsdauer, da immer mehr Workflows, Automatisierungen und Prozesse auf die spezifische Plattform zugeschnitten werden.

## 3. Umgang mit Lock-in-Effekten

Lock-in-Effekte lassen sich nie vollständig vermeiden, aber sie können bewusst gemanagt werden. Die wichtigste Strategie ist die Einführung von Abstraktionsschichten. In unserem Fall könnte die CI/CD-Logik in plattformunabhängigen Shell-Skripten gekapselt werden, die von jedem CI-System aufgerufen werden können. Statt die Testbefehle direkt in den GitHub Actions Workflow zu schreiben, würde ein zentrales Skript existieren, das sowohl von GitHub Actions als auch von GitLab CI oder Jenkins ausgeführt werden kann.

Eine weitere wichtige Strategie ist die konsequente Wahl von Standards statt Produkten. Git statt proprietärer Versionsverwaltung, Docker-Container statt plattformspezifischer Deployments, Markdown statt herstellereigener Dokumentationsformate. Je mehr standardisierte Technologien eingesetzt werden, desto geringer ist der Migrationsaufwand bei einem Wechsel.

Unternehmen sollten außerdem eine Exit-Strategie für jede proprietäre Komponente dokumentieren. Diese Strategie beantwortet die Frage: Wie würde eine Migration konkret aussehen, wie lange würde sie dauern und welche Kosten würden entstehen? Bereits die Beantwortung dieser Fragen schafft Transparenz über die tatsächliche Abhängigkeit. Ergänzend dazu sollten regelmäßige Evaluierungen stattfinden — mindestens jährlich sollte geprüft werden, ob die aktuellen Tools noch die beste Wahl darstellen oder ob offenere Alternativen verfügbar sind.

Nicht zuletzt ist die Sicherstellung der Datenportabilität entscheidend. Regelmäßige Backups der Issues und Projects per API-Export, Dokumentation in universellen Formaten und Artefakte, die nicht ausschließlich bei einem Anbieter gespeichert sind, reduzieren die Abhängigkeit erheblich.

## 4. Verallgemeinerte Betrachtung

Die Entscheidung zwischen Open-Source und proprietärer Software ist keine binäre Wahl, sondern eine kontextabhängige Abwägung. Für Infrastruktur und Entwicklungstools hat sich Open-Source in den letzten zwei Jahrzehnten als Standard etabliert. Linux betreibt den Großteil aller Server weltweit, PostgreSQL und MySQL dominieren den Datenbankmarkt, und Programmiersprachen wie Python, Java und JavaScript sind allesamt quelloffen. In diesen Bereichen bietet Open-Source nicht nur Kostenvorteile, sondern auch eine höhere Anpassbarkeit und Unabhängigkeit.

Proprietäre Software hat hingegen ihre Berechtigung bei spezialisierten Branchenlösungen, wo der Aufwand einer Eigenentwicklung den Nutzen bei weitem übersteigen würde. SAP für Unternehmensressourcenplanung, spezialisierte CAD-Software für Ingenieure oder branchenspezifische Compliance-Tools sind Beispiele, in denen proprietäre Lösungen oft die pragmatischere Wahl darstellen. Auch wenn professioneller Support mit garantierten Reaktionszeiten (SLAs) und vertraglicher Haftung erforderlich ist, kann proprietäre Software die sicherere Option sein.

Für unser Berufsschulprojekt war die Entscheidung eindeutig: Mit Ausnahme von GitHub sind alle eingesetzten Technologien Open-Source. Dies minimiert Kosten, maximiert die Lernmöglichkeiten (da der Code aller Tools einsehbar ist) und vermeidet unnötige Abhängigkeiten. Der Lock-in bei GitHub wird bewusst in Kauf genommen, da die Vorteile (integriertes Ökosystem, kostenloser CI/CD-Dienst, breite Verbreitung in der Industrie) die Risiken für ein Projekt dieser Größe überwiegen.

## 5. Proprietäre Software und Demokratie

Die zunehmende Digitalisierung aller Lebensbereiche verleiht der Frage, wer die Software kontrolliert, eine gesellschaftspolitische Dimension. Wenn Wahlsoftware, Behördensysteme oder kritische Infrastruktur auf proprietärer Software basiert, kann die Öffentlichkeit nicht überprüfen, ob diese Systeme korrekt, fair und sicher funktionieren. Diese Intransparenz steht im Widerspruch zu demokratischen Grundprinzipien wie Nachvollziehbarkeit und öffentlicher Kontrolle.

Ein konkretes Beispiel ist der US-amerikanische CLOUD Act, der US-Behörden den Zugriff auf Daten ermöglicht, die bei US-Unternehmen gespeichert sind — unabhängig davon, ob die Server physisch in Europa stehen. Wenn europäische Behörden und Unternehmen ihre sensiblen Daten auf Plattformen wie Microsoft Azure, Google Cloud oder Amazon AWS verarbeiten, entsteht eine Abhängigkeit, die nicht nur wirtschaftlich, sondern auch geopolitisch problematisch ist. Im Falle von Sanktionen oder politischen Konflikten könnte der Zugang zu diesen Diensten eingeschränkt oder entzogen werden.

Die Monopolbildung im Technologiesektor verschärft dieses Problem. Wenige große Konzerne — Microsoft, Google, Apple, Amazon — dominieren den Markt für Betriebssysteme, Cloud-Dienste, Produktivitätssoftware und mobile Plattformen. Diese Konzerne können Preise, Funktionen und Datenschutzrichtlinien einseitig diktieren, während Nutzer aufgrund der hohen Wechselkosten kaum Alternativen haben.

Um mit dieser Situation aufgeklärt umzugehen, sind Maßnahmen auf allen Ebenen erforderlich. Unternehmen sollten eine Open-Source-First-Strategie verfolgen, ihre Anbieter diversifizieren und Mitarbeiter in offenen Technologien schulen. Der Staat sollte dem Prinzip "Public Money, Public Code" folgen und Software, die mit öffentlichen Mitteln entwickelt wird, als Open-Source bereitstellen. Die Förderung europäischer Initiativen wie GAIA-X für eine souveräne Cloud-Infrastruktur ist ein Schritt in die richtige Richtung. Für Privathaushalte geht es vor allem um Bewusstsein: Die Abhängigkeit von geschlossenen Ökosystemen erkennen, Open-Source-Alternativen wie LibreOffice, Firefox oder Signal nutzen und auf Datenportabilität achten — also die eigenen Daten regelmäßig exportieren und nicht ausschließlich in proprietären Formaten speichern.

Die Frage, ob proprietäre Software eine Bedrohung für die Demokratie darstellt, lässt sich nicht pauschal beantworten. Sie ist es dann, wenn sie in Bereichen eingesetzt wird, in denen Transparenz und öffentliche Kontrolle unverzichtbar sind. Die Lösung liegt nicht in der Abschaffung proprietärer Software, sondern in einer bewussten, diversifizierten und souveränen Nutzung digitaler Technologien.
